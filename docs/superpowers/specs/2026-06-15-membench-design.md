## 1. Overview & Goal

**membench** is a benchmark harness that measures the **Minni** local-first agent-memory system against a set of RAG / agent-storage baselines, on the operator's own real corpus, across **four metric families**:

1. **Retrieval latency** — how long a query takes to return context (p50 / p95 wall-clock).
2. **Retrieved-context token cost** — how many tokens of context each system spends to answer.
3. **Retrieval quality** — recall@k, precision@k, nDCG@k, MRR, and correct-refusal rate on negatives.
4. **End-task success** — whether an agent, fed *only* that system's retrieved context, actually answers the user's question correctly.

The goal is **not** to produce a marketing number. It is to produce a number that **survives a hostile methodology review** — a benchmark the operator can publish and defend when a skeptic says *"isn't this rigged in Minni's favor?"*. Every design decision below is made so that the answer to that question is provably *"no, and here is the control that prevents it."*

**Primary deliverable of the harness (not this doc):** a reproducible report, regenerated by **one command from a clean checkout**, that reports per-adapter scores on all four families with pinned corpus hash, model ids, and tokenizer, plus an honest threats-to-validity section carried into the report itself.

---

## 2. The Claim Being Tested (and the honest caveat)

**Claim under test (falsifiable):**

> For an agent operating over a personal local knowledge vault, Minni delivers **better task success per token of retrieved context** and **higher retrieval quality** (including correctly refusing to answer when the vault genuinely lacks the fact) than naive RAG and the common local-memory archetypes — *while paying a measurable, fully-counted latency tax for its governance hop.*

**The honest caveat, stated up front so a reviewer cannot "catch" us with it:**

Minni's thesis is **token-efficiency + retrieval quality + task success**. It is **not** raw vector-retrieval speed. Minni routes every recall through a real daemon hop and a provenance/governance gate. That extra work **will likely make latency Minni's *weakest* axis** — on p50/p95 latency we *expect* naive in-process FAISS to beat Minni, possibly by a wide margin. We report that loss in full. A benchmark that hid the latency tax would be the rigged one; membench shows it.

**What would falsify the claim** (so it is not unfalsifiable):
- If naive_rag matches or beats Minni on **task-success-per-token** at equal token budget, the token-efficiency claim fails.
- If naive_rag matches or beats Minni on **nDCG@k / recall@k** on multi-hop and contradiction bands, the retrieval-quality claim fails.
- If Minni's correct-refusal rate on negatives is **not** higher than the ungoverned baselines, the governance claim fails.
- Each of these is a concrete, pre-registered number, not a vibe.

## 3. Architecture

membench is a **two-layer harness**. Layer 1 is the scrutiny floor (deterministic, cheap, CI-gated). Layer 2 is the payoff (agent-in-the-loop, nondeterministic, reported with variance).

### 3.1 The MemoryAdapter contract

Every system under test — Minni and every baseline — is wrapped in **one identical interface**. The harness only ever talks to this contract; it never reaches into a system's internals.

```python
class MemoryAdapter(Protocol):
    name: str                     # stable id: "minni", "naive_rag", ...
    config_hash: str              # hash of this adapter's pinned config (see §7)

    def ingest(self, corpus: FrozenCorpus) -> IngestReport:
        """Load the IDENTICAL frozen corpus bytes. Build whatever index the
        system natively builds. Returns build time, index size, doc count.
        MUST receive only the scrubbed FrozenCorpus object (§5.1) — never raw
        vault bytes."""

    def query(self, q: str, budget: TokenBudget) -> QueryResult:
        """Answer a single retrieval query under a token budget. `q` is
        pre-validated by the harness (see Input validation below) before it
        ever reaches an adapter."""

    def teardown(self) -> None:
        """Release resources; leave no state that could leak into the next run."""
```

**Supporting types (informal schema — full field validation lands in `contract.py`, s1):**

```python
@dataclass(frozen=True)
class TokenBudget:
    max_tokens: int                   # hard cap on tokens in context_string — RUNTIME-ENFORCED:
                                      # both runner_layer1.py and runner_layer2.py assert
                                      # harness_tokens <= budget.max_tokens immediately after
                                      # computing harness_tokens from the returned QueryResult,
                                      # and ABORT the run on violation (no silent truncation).
    max_docs: int                     # hard cap on len(ranked_results); == k by default (§6)

@dataclass(frozen=True)
class RankedDoc:
    doc_id: str                       # canonical whole-file doc-id (see Doc-id granularity below)
    score: float                      # adapter's own relevance score (monotone, higher = better);
                                      # used only for ordering + diagnostics, NOT for metric math

class FrozenCorpus(Protocol):
    content_hash: str                 # SHA-256 over the canonical sorted per-file manifest (§5.1)
    scrubbed: bool                    # MUST be True; corpus.py asserts before instantiation (§5.1)
    def doc_ids(self) -> list[str]: ...        # all canonical whole-file doc-ids. Construction MUST
                                               # apply realpath-containment to EVERY discovered path,
                                               # filtering or aborting on any path whose realpath
                                               # escapes corpus_dir (closes the symlink-in-corpus gap
                                               # that would otherwise survive doc_ids() and only be
                                               # caught at read()).
    def read(self, doc_id: str) -> bytes: ...  # scrubbed bytes for one doc.
                                               # MUST validate doc_id ∈ doc_ids() (membership)
                                               # AND that realpath(corpus_dir/doc_id) stays under
                                               # realpath(corpus_dir) BEFORE opening any file —
                                               # path-traversal guard (see below).

@dataclass(frozen=True)
class IngestReport:
    build_wall_clock_ms: float        # time to build the adapter's native index
    index_size_bytes: int             # on-disk / in-memory index footprint
    doc_count: int                    # MUST equal the number of source corpus FILES processed from
                                      # FrozenCorpus (one per corpus.doc_ids() entry), NOT the number
                                      # of internal chunks produced. Chunk counts are internal
                                      # implementation details.
    ingest_tokens_used: int           # generation tokens spent at ingest (0 for non-LLM adapters;
                                      # populated for llm_wiki curation — see §6.8)

@dataclass(frozen=True)
class QueryResult:
    ranked_results: list[RankedDoc]   # ordered, DEDUPLICATED doc-ids + scores (see granularity below)
    context_string: str               # the exact CONTENT-ONLY text that would be handed to the model
    wall_clock_ms: float              # measured end-to-end inside query(), incl. all hops
    refused: bool = False             # adapter MUST set True when it explicitly declines to answer
                                      # (e.g. Minni's gate refuses on insufficient provenance).
                                      # Used by §6.5 scoring, but ONLY honored when ranked_results
                                      # is also empty: refused=True with a non-empty ranked list is
                                      # NOT a refusal (see §6.5 correct-refusal predicate).
```

Rules baked into the contract:
- `ranked_results` are **canonical whole-file doc-ids from the frozen corpus**, enabling Layer-1 quality scoring against gold docs.
- `context_string` is **exactly** what Layer 2 will feed the agent — no separate "nice" version. It is **content-only text**: adapters MUST NOT emit system-role markers, instruction preambles, or anything other than retrieved corpus content. **"System-role marker" has a concrete, testable definition** — a banned-pattern list pinned in `contract.py` as `BANNED_ROLE_MARKERS` (case-insensitive, matched anywhere in `context_string`): `SYSTEM:`, `ASSISTANT:`, `HUMAN:`, `USER:`, `<|system|>`, `<|assistant|>`, `<|user|>`, `<|im_start|>`, `<|im_end|>`, `<system>`, `</system>`, and the `<retrieved_context` boundary tag itself. The §9.4 conformance test imports this exact list as its oracle, so every adapter is checked against the same set (no implementer guesses what counts as a marker). The Layer-2 runner wraps the content in an untrusted-data boundary (see Prompt-injection boundary below).
- `wall_clock_ms` is measured **inside** `query()`, so Minni's daemon hop and gate are *inside the timed region* (see Fairness §7).
- **Token counting is harness-owned, not adapter-owned.** `QueryResult` has **no** `tokens_used` field. After `query()` returns, the harness computes the authoritative token count by tokenizing `context_string` with the canonical tokenizer (§6.6) and attaches it as `harness_tokens` to the scored record. An adapter therefore *cannot* under-count: there is no adapter-supplied token field to deflate. **Runtime budget enforcement (load-bearing):** immediately after computing `harness_tokens`, **both** `runner_layer1.py` and `runner_layer2.py` assert `harness_tokens <= budget.max_tokens` and **ABORT the run** on violation — the over-budget result is **never** silently truncated to fit, so an adapter cannot smuggle extra context past the cap. The §9.4 conformance test asserts the harness count is present and equals `len(canonical_tokenizer.encode(context_string))`, and that an over-budget adapter trips the abort.

**Doc-id granularity (resolves chunk-vs-doc scoring).** The canonical retrieval unit for *all* metric math is the **whole file** (one `doc_id` per corpus file). Chunk-level adapters (`naive_rag`, the semantic pass of `markdown_grep`, `llm_wiki`) retrieve chunks internally but MUST **collapse chunks to parent whole-file doc-ids before returning**, using **first-hit deduplication**: walk the chunk ranking top-down, emit each parent `doc_id` the first time it appears, drop later chunks of an already-emitted doc. This guarantees `ranked_results` contains **no duplicate `doc_id`** (so precision@k and nDCG@k are well-defined) and that "10 chunks from 1 doc" collapses to a single ranked doc rather than masquerading as 10 hits. The §9.4 conformance test asserts `ranked_results` has unique `doc_id`s drawn only from `corpus.doc_ids()`.

**Input validation (harness-side, before dispatch).** Before any adapter sees `q`, the harness validates it: **non-empty, ≤ 512 bytes UTF-8, no null bytes, valid Unicode**. A query failing validation aborts the run with a clear error (gold files are author-controlled, so this is a corruption tripwire, not a runtime filter). Adapters never receive an un-validated query. `markdown_grep` (and any subprocess-backed adapter) MUST pass `q` to its search tool as a **positional argument via list-form `subprocess` invocation** (`["rg", "--", q, path]`) — **never** `shell=True` with `q` interpolated — so a regex/metacharacter/injection payload in `q` cannot reach a shell. This is mandated in the adapter spec (§4.1) and enforced by the **subprocess-safety lint (§9.10)**, which statically asserts that **no** file under `bench/` (not just `markdown_grep`) ever calls `subprocess.*` with `shell=True`. The lint runs over all adapter files so a future adapter or a maintenance edit cannot reintroduce a shell-injection path.

**Prompt-injection boundary (Layer 2).** Because `context_string` is fed to the agent-under-test, the Layer-2 runner treats it as **untrusted external data**: it wraps the string in a delimited block and the fixed system prompt instructs the agent that the block is retrieved data, not instructions, and must never be obeyed as commands. Two concrete hardening rules close the obvious delimiter-breach hole (a corpus doc that contains the literal closing tag could otherwise escape the boundary):
1. **Per-run nonce delimiter.** The boundary tag carries a random per-run nonce — `<retrieved_context id="{nonce}"> … </retrieved_context id="{nonce}">` where `nonce` is freshly generated each run and is **not** derivable from corpus content. The system prompt names the exact nonce as the only legitimate boundary. A static `</retrieved_context>` embedded in a vault doc cannot match the nonce'd closer, so it cannot forge the end of the trusted boundary.
2. **Escape literal occurrences.** Before wrapping, the runner additionally escapes any literal `<retrieved_context` / `</retrieved_context` substrings found inside `context_string` (e.g. replacing `<` with a sentinel), so injected markup cannot even resemble the delimiter.

Combined with the content-only constraint on `context_string` above, this prevents a buggy or poisoned adapter from injecting judge-friendly directives (e.g. "output SUCCESS") that would corrupt Layer-2 scoring for the whole run. This is **standard, documented-to-be-imperfect** RAG prompt-injection mitigation: it raises the bar (nonce + escaping + content-only + instruction) but does **not** prove the boundary unbreakable — a sufficiently crafted in-context payload can still influence a model. The **residual risk is named explicitly in Threats to Validity (§8.9)**; the corpus here is the operator's own (presumptively non-adversarial) vault, which bounds — but does not eliminate — the exposure.

### 3.2 Layer 1 — offline, deterministic, no LLM (runs in CI)

For each adapter:
1. `ingest(frozen_corpus)` — same bytes for all.
2. Replay the full labeled query set through `query(q, budget)`.
3. Score, **without any LLM**: latency (p50/p95), token cost, and retrieval-quality metrics (recall@k, precision@k, nDCG@k, MRR) against the gold doc set, plus **correct-refusal rate** on negative queries.

**Determinism requirement (hard):** a re-run on the *same pinned corpus content-hash + same adapter configs + same tokenizer + same pinned runtime* MUST produce **byte-identical** Layer-1 scores. Any nondeterministic source (ANN index randomness, thread-scheduling-dependent tie-breaks, unset embedding seeds) must be pinned or removed. Because nDCG and MRR involve `log2` and floating-point division, **bit-level float results can differ across Python or math-library versions** — so byte-identity additionally requires pinning the **runtime**: the **Python version** and the versions of **numpy / scipy** (and any other float-math library used by `metrics.py`) are pinned in a committed lockfile (`bench/pyproject.toml` + a `bench/requirements.lock` produced by `pip-compile` or `uv lock`; see §11) and recorded in the report header (§7.7). CI runs Layer-1 determinism on the pinned interpreter; a runtime change that perturbs float bits is a *config* change (new lockfile hash), not a silent benchmark-logic failure. Layer 1 is the part the skeptic can run themselves and get the same numbers — that is its whole point. CI fails the build if two consecutive Layer-1 runs on the pinned runtime disagree.

*Note on latency vs determinism:* latency numbers are **wall-clock and therefore machine-dependent** — they are reported but are explicitly **excluded from the byte-identical determinism gate**. The determinism gate covers only the *scores derived from ranked_results and token counts* (recall@k, precision@k, nDCG@k, MRR, token cost, correct-refusal). **The exclusion is by an explicit allow-list of stripped field names, not a blanket drop:** the determinism comparison strips **exactly** `wall_clock_ms` and `build_wall_clock_ms` (the timing fields named in §3.1) and asserts **every other field byte-identical**. The stripped set is a named constant `DETERMINISM_EXCLUDED_FIELDS = {"wall_clock_ms", "build_wall_clock_ms"}` in `metrics.py`; the §9.1 test asserts this set is small and that at least the score fields survive the strip (so a lazy implementer cannot strip the whole record and trivially pass). Latency is reported as a distribution with the host spec recorded; CI checks latency only for gross regression, not byte-identity. This split is called out so a reviewer does not read "byte-identical scores" and then catch us on jittery timings.

### 3.3 Layer 2 — agent-in-the-loop, nondeterministic

1. For each scripted episode, feed an agent **only** the adapter's `context_string` for each turn.
2. The agent answers.

**Inter-turn context policy (load-bearing — chosen and fixed here):** the agent's context window is **reset to the fixed shared system prompt between every turn**. Each turn is a **fresh session**: the only episode-specific content the agent sees on a given turn is that turn's question plus the adapter's freshly-retrieved `context_string` for that question, wrapped in the `<retrieved_context>` boundary. **No prior turn's question, answer, or retrieved context is carried into the agent context.** This is the policy that actually tests the retrieval system's job: bridging sessions. If context instead accumulated across turns, an early turn would leak the answer to a later turn and *every* adapter would benefit equally from that in-context history, making the cross-session retrieval the episodes are designed to test invisible. Because the context resets, the harness **re-retrieves via `query()` for every turn** — there is no shared retrieval across turns within an episode. The episode's "multi-session" character lives entirely in the *corpus* (a fact authored in a doc dated to an earlier session, queried by a later turn — see episode write-path below), not in any accumulated agent state.
3. A **calibrated LLM judge** scores answer correctness + task success against the gold facts.
4. Count **tokens-to-model** (the actual prompt tokens the agent consumed, canonical tokenizer).
5. Run **N trials** (default N=5, pinned in config) per episode per adapter; report **mean and variance / 95% CI**, never a single lucky run.

**Episode count is fixed early because it is load-bearing later:** membench uses **15–25 scripted multi-session episodes** (target 20). This number is established here in s0 so downstream slices (episode authoring s4, judge calibration s5, reporting s7) size against it. Episodes are *multi-session* (a fact stated in session 1, queried in session 3) precisely to exercise the thing Minni claims to be good at; a single-session benchmark would under-test all memory systems equally and is explicitly out of scope.

**Episode write-path (resolves the stateless-adapter fairness question).** In s0/s1 the multi-session episodes test **recall of facts already present in the frozen corpus** — the "session 1 wrote it" framing means *the fact lives in a corpus doc dated to an earlier session*, not that the agent must persist a brand-new write at runtime. The frozen corpus is pre-ingested once; every adapter retrieves against the same pre-ingested bytes, so no adapter is structurally disadvantaged by lacking a runtime write path. **In-session writes (the agent persisting a fact mid-episode and retrieving it in a later turn) are explicitly OUT OF SCOPE for the current 8-slice plan** and are recorded as future work (§10), because they would advantage Minni's write/governance path against stateless adapters and require a separate fairness design. Any episode that would require a runtime write is rejected at authoring (s4). This keeps "multi-session" a real recall-over-time test (most-recent-corpus-doc-wins, contradiction/correction bands) without smuggling in a write-path bias.

**Judge gates (hard, all required for Layer-2 numbers to count):**
- **Calibration gate (both thresholds required, on a minimum-size subset):** against a **human-checked subset** of judgments scored by the operator, the judge must clear **BOTH** (a) **raw agreement ≥ 0.80** AND (b) **Cohen's κ ≥ 0.60** (the "substantial" threshold on the Landis–Koch scale). **The human-checked subset MUST contain ≥ 40 paired judgments.** This minimum is mandatory because κ's standard error scales as ≈ √(p_o(1−p_o)/n): at n=20 the 95% CI on κ spans roughly ±0.22–0.44, so a measured κ=0.62 could have a true value near 0.20 — a judge barely above chance would pass. At n ≥ 40 the gate is credibly tight. A calibration run with n < 40 paired judgments is **rejected with a hard error** (not a warning) and its judge numbers may not be published; the harness records n alongside the reported κ and raw agreement. Both numbers are reported. The κ requirement is mandatory because raw agreement is **chance-inflated on skewed labels**: if 85% of episodes are "success", a judge that always outputs "success" scores 0.85 raw agreement yet κ = 0 (no better than chance) — the κ ≥ 0.60 gate fails that degenerate judge where a raw-agreement-only gate would have passed it. If the judge misses *either* threshold, its rubric is revised and recalibrated; un-calibrated judge output is not published.
- **No-self-judge gate (concretely defined `model_family`):** the **judge model MUST differ in family from the agent-under-test model**. "Family" is **not** organization/provider; it is an explicit config field. Each model in config carries `model_family: str` (e.g. `claude-opus`, `claude-sonnet`, `gpt-4-class`, `o-series`, `gemini`, `llama`). CI rejects any config where `agent_model.model_family == judge_model.model_family`. A different *version* within the same family (e.g. two Claude Opus point releases) does **not** satisfy the gate; the family strings must differ. The pair (agent_model_id+family, judge_model_id+family) is pinned in config and printed in the report header. The initial family table for expected candidates is maintained in `config.py` and reviewed at s5.

## 4. Adapters

Five adapters implement the contract. Two are **archetypes** representing common local patterns — they are deliberately *faithful reconstructions of a documented pattern*, **not** product installs, and they are labeled as such in the report.

| Adapter | What it is | Governance? | Representing |
|---|---|---|---|
| `minni` | Recall through the **real** Minni daemon + provenance gate | Yes (the point) | Minni itself |
| `naive_rag` | chunk → embed → top-k over FAISS or Chroma | No | The textbook RAG baseline |
| `native_platform` | Claude Code session memory / context compaction | Partial/implicit | The "just use the platform's built-in memory" position |
| `markdown_grep` | Files-on-disk + lexical + semantic search | No | The **Obsidian-vault-as-agent-memory** archetype |
| `llm_wiki` | LLM-curated interlinked wiki + retrieval | No | The **Karpathy llmWiki** archetype |

### 4.1 Landscape survey (why the archetypes are faithful, with citations)

The two archetypes are the ones most open to the objection *"that's a strawman you built to lose."* They are not. Each reconstructs a **documented, widely-adopted community pattern**:

**`markdown_grep` represents Obsidian-as-agent-memory.** The dominant local-first agent-memory pattern in 2026 is a plain-Markdown Obsidian vault that the agent reads and writes directly: *"Obsidian is nothing more than a folder on your disk, and Claude Code knows how to read text files — no magic or complicated integration needed."* The pattern is wired to agents either via the Obsidian Local REST API plugin or an Obsidian MCP server that exposes the vault as MCP tools to any MCP-aware client. Its explicit selling point against vector stores is *"no embedding cliff, no provider lock-in, no opaque store"* — i.e. lexical/path search over real files. `markdown_grep` reconstructs exactly this: the frozen corpus laid out as files, answered by lexical search (ripgrep-class) **plus** an optional semantic pass, with **no governance layer**. The lexical search MUST invoke ripgrep via **list-form `subprocess` with `q` as a positional argument after `--`** (`["rg", "--fixed-strings", "--", q, corpus_dir]` or an equivalent regex mode with `q` still positional) and **never** `shell=True` with `q` interpolated, so a metacharacter/regex/injection payload in `q` cannot reach a shell (the harness already length-bounds and null-checks `q` per §3.1). The no-`shell=True` rule is machine-enforced for every file under `bench/` by the subprocess-safety lint (§9.10), not left to prose. Sources:
- Pasquale Pillitteri, *"Obsidian + Claude Code: The Second Brain That Makes AI Agents Actually Useful"* — https://pasqualepillitteri.it/en/news/962/obsidian-claude-code-second-brain-persistent-memory
- NxCode, *"Obsidian AI Second Brain: Complete Guide (2026)"* — https://www.nxcode.io/resources/news/obsidian-ai-second-brain-complete-guide-2026
- *Open Second Brain* MCP server (deterministic plain-Markdown memory layer inside an Obsidian vault) — https://mcpserver.space/mcp/open-second-brain/

**`llm_wiki` represents Karpathy's llmWiki.** Andrej Karpathy published the LLM-Wiki pattern as a GitHub gist (April 2026): instead of RAG over raw docs, an LLM agent **incrementally builds and maintains a structured, interlinked Markdown wiki** — *"Obsidian is the IDE; the LLM is the programmer; the wiki is the codebase."* The defining property vs RAG: *"an LLM wiki pre-compiles knowledge into structured, interlinked entity pages — so answers compound over time instead of being rediscovered from scratch."* `llm_wiki` reconstructs this: an LLM-curated set of entity pages built from the frozen corpus, then retrieved at query time — again with **no governance/provenance gate**.

**`llm_wiki` ingest constraints (fairness-critical — see §7.10, §7.11).** The curation LLM that builds the wiki pages is a **generation** model, not an embedding model, so it is *not* covered by the shared-embedder control (§7.2); it gets its own pinned control. The curation model is **pinned in config** (`llm_wiki_curation_model_id` + `model_family`), **printed in the report header**, and — because "stronger" has no single operational definition (benchmark score, parameter count, price tier, and vendor capability class all disagree) — the rule is the one criterion the spec can actually enforce: **the curation model MUST use the same `model_id` as the agent-under-test.** CI rejects any config where `llm_wiki_curation_model_id != agent_model_id`. (Optional future relaxation, not enabled now: permit an "equivalent-or-weaker" model only if it shares the agent's `model_family` or appears at a *lower* tier in a maintained tier table in `config.py`; until that table exists, same-`model_id` is the hard rule.) This guarantees `llm_wiki` cannot enter the benchmark with knowledge pre-distilled by a frontier model no other adapter had access to. The curation step **ingests only the scrubbed `FrozenCorpus`** (never raw vault bytes — §5.1), so it cannot echo secrets into generated wiki pages. The generation tokens spent during curation are recorded in `IngestReport.ingest_tokens_used` and surfaced per §6.8. Sources:
- Karpathy, llm-wiki gist — https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f
- Data Science Dojo, *"LLM Wiki by Andrej Karpathy: Build a Compounding Knowledge Base (Tutorial)"* — https://datasciencedojo.com/blog/llm-wiki-tutorial/
- MindStudio, *"What Is the Karpathy LLM Wiki Pattern?"* — https://www.mindstudio.ai/blog/karpathy-llm-wiki-knowledge-base-pattern

**Where Mem0 / Letta / Zep fall among these archetypes (and why they are not separate adapters in s0):** the three leading commercial/OSS agent-memory frameworks each map onto a point our archetypes already bracket on the quality axes membench measures:
- **Mem0** — a vector-first memory layer with an extract-then-ADD/UPDATE/DELETE/NOOP reconciliation step; on the retrieval axes it behaves like a *curated* `naive_rag` (vector recall, recency/similarity-strong, multi-hop-weak).
- **Zep (Graphiti)** — a *temporal knowledge graph* with validity windows ("true from X until Y"); its differentiator lands squarely in our **recency-sensitive** and **contradiction/correction** difficulty bands, which is exactly why those bands exist.
- **Letta (MemGPT)** — an OS-style core/recall/archival tiering where the agent pages its own memory; architecturally closest to `native_platform` (platform-managed context) plus an archival store.
Source: Wasowski, *"AI Agent Memory 2026 — Comparing Mem0, Zep, Graphiti, Letta, LangMem"* — https://medium.com/@wasowski.jarek/i-compared-5-ai-agent-memory-systems-across-6-dimensions-none-wins-6a658335ed0a ; Mem0, *"Graph-Based Memory Solutions for AI Agents"* — https://mem0.ai/blog/graph-memory-solutions-ai-agents

These three are recorded as **first-class candidates for a later slice** (post-s7), once the contract is proven against the archetypes. Adding them now is scope creep; bracketing them with archetypes is the honest minimum. This is an *open question*, see §10.

## 5. Corpus & Labeling Protocol

### 5.1 The frozen corpus

- **Source:** a snapshot of the operator's **real Minni vault** (~770 docs). Real corpus, because a synthetic corpus would not test the messy reality (typos, contradictions, half-finished notes, recency) the systems must survive.
- **Frozen & content-hashed:** the snapshot is captured once, **content-hashed** (SHA-256 over a canonical sorted manifest of per-file hashes), and pinned. Every adapter ingests **identical bytes**. The corpus hash is recorded in every report.
- **Scrub-before-instantiation gate (no adapter ever sees raw bytes).** Secrets are **scrubbed** before snapshotting (a scrub pass redacts tokens/keys/emails; the scrub manifest is reviewed by the operator). This is enforced structurally, but **not** by a self-asserted boolean alone: a bare `scrubbed == True` attribute on a structurally-typed `FrozenCorpus` Protocol implementor could be set over un-scrubbed bytes and bypass the gate. So the scrub gate is **cryptographically bound**:
  - The scrub tool emits, alongside the corpus, a **`scrub_manifest_hash`** = SHA-256 over the canonical sorted list of redacted spans **and** the resulting scrubbed file tree. This hash is produced **only** by the scrub tool's output, not by an adapter.
  - `corpus.py` **does not accept an arbitrary `FrozenCorpus` Protocol implementor** for a real run: it **loads the corpus only from the directory the scrub tool wrote**, and requires **both** `content_hash` and `scrub_manifest_hash` to be present and to **cross-check** (the loader recomputes both over the on-disk scrubbed tree and aborts on any mismatch). `scrubbed == True` is necessary but not sufficient; the two recomputed hashes are the real gate. A compliant `content_hash` over raw vault bytes fails because no matching `scrub_manifest_hash` exists for them.
  - Because every adapter — including `llm_wiki`'s curation LLM and `naive_rag`'s embedder — receives only this hash-verified scrubbed `FrozenCorpus`, no adapter can ingest un-scrubbed bytes regardless of call order, and no secret can be echoed into a generated wiki page, an embedding, or a `context_string` recorded in a report artifact.
  - **Defense in depth on log exposure:** report artifacts **never** log a raw `context_string` longer than a fixed truncation limit (`CONTEXT_LOG_TRUNCATE` in `config.py`, default 256 chars, with the remainder elided), so even if scrubbing missed a secret, the blast radius in a committed/shared report is bounded. This truncation is **not** advisory or `report.py`-local (which any earlier log path could bypass): `FrozenCorpus.read()` output and `QueryResult.context_string` are **NEVER** written to any logger at DEBUG level or above **except through a single log-safe wrapper** that enforces `CONTEXT_LOG_TRUNCATE`. All code under `bench/` routes these values through that one wrapper; direct logging of either value is prohibited. A §9.12 test injects a 10,000-char `context_string` and asserts no emitted log line exceeds `CONTEXT_LOG_TRUNCATE` plus a small margin. Full `context_string`s, when needed for debugging, are written only under `_private/` (gitignored).
  - `scrub_manifest_hash` is pinned and printed in the report header (§7.7).
- **Path-traversal guard on `corpus.read()` (no adapter can read arbitrary host files).** A buggy or malicious adapter could return a `doc_id` like `../../etc/passwd` or an absolute path. `corpus.read(doc_id)` therefore **MUST**, before resolving any path: (1) assert `doc_id` is a member of the pre-built `doc_ids()` set, and (2) assert that `os.path.realpath(os.path.join(corpus_dir, doc_id))` has `os.path.realpath(corpus_dir)` as a prefix (containment check). Any `doc_id` failing either check raises, never opens a file. **The same realpath-containment is also applied at `doc_ids()` construction time** (§3.1): every path discovered while enumerating the corpus is realpath-resolved and any whose realpath escapes `realpath(corpus_dir)` is filtered out or aborts construction — so a symlink planted inside the corpus that points outside it never even enters the `doc_ids()` set, closing the gap that a `read()`-only guard would leave open until call time. This invariant is independent of the §9.4 adapter-output check (which validates `ranked_results` membership): `read()` defends itself even if a malformed gold-label file or a buggy adapter slips a bad id through. A **path-traversal negative test** lives in `test_corpus_integrity.py` (§9.5).
- **Privacy:** the **private corpus and the raw labels stay OUT of public git** — they live under the already-gitignored `_private/` tree. This is a **machine-checked invariant**, not a static claim: a CI test (`test_private_gitignored.py`, §9.8) runs `git check-ignore` on representative `_private/` paths and fails if any are not ignored; a **pre-commit hook** (specified in §11) rejects any commit that stages a path under `_private/`; and `bench/pyproject.toml` excludes `_private/` from packaging.

  **Pre-commit install step (the hook is useless if it never runs).** `pre-commit` reads its config from the **repo root**, not from a subdirectory, so a `.pre-commit-config.yaml` sitting only under `bench/` would be silently skipped by a contributor running `git commit` at the root — defeating the machine-checked invariant. Therefore: the membench `_private/`-rejecting hook is added as a hook **block in the repo-root `.pre-commit-config.yaml`** (additive to whatever the Minni repo already configures there; if the repo has no root config, `make setup` creates one). The `bench/Makefile` `setup` target (and `bench/README.md`) run `pre-commit install` at the repo root so the hook is actually registered into `.git/hooks`. The `bench/.pre-commit-config.yaml` referenced in §11 is the **source fragment** that is merged into / referenced from the root config, not a standalone config expected to run on its own. This is also covered in s1 acceptance: `git commit` of a staged `_private/` path must be rejected from a clean checkout after `make setup`. **Because the local hook is developer opt-in (`make setup` may never be run), it is backstopped by a server-side control: CI itself runs a check that no `_private/` path appears in `HEAD` or in any PR diff**, and fails the build if one does. This server-side gate is independent of developer opt-in and complements — does not replace — the local pre-commit hook. The public repo ships the *harness*, the *schemas*, the *metric code*, and a **small synthetic fixture corpus** so outsiders can run the pipeline end-to-end without the operator's private data.

### 5.2 Gold set (ground truth)

Ground truth is a set of **(question → gold-doc(s) → gold-fact triples)**:
- **question** — natural-language query.
- **gold-doc(s)** — the corpus doc-id(s) that genuinely contain the answer (≥1; empty set for negatives).
- **gold-fact** — the atomic fact(s) the correct answer must assert, used by the Layer-2 judge.

**Authoring protocol (agent-drafted, human-approved — this ordering is mandatory):** an agent *drafts* candidate triples from the corpus; the **operator reviews, corrects, and approves** each one. Drafts are never promoted to gold without explicit operator approval. Approved labels carry an `approved_by` + timestamp. This keeps labels cheap to produce *and* trustworthy — and it is recorded here so no future contributor "helpfully" auto-promotes unreviewed drafts.

**Negative-authoring sub-protocol (mandatory — a mis-labeled negative corrupts refusal scoring).** Labeling a query negative (`G(q) = ∅`) asserts that **no** doc in the ~770-doc vault answers it — a far stronger, harder-to-verify claim than citing a supporting doc for a positive. A query becomes a gold negative **only if all of the following fail to find an answer**: (1) the drafting agent searches **exhaustively** — lexical (ripgrep over the full vault) **and** semantic (top-k vector recall against the pinned embedder) — and finds no supporting doc; (2) the **operator independently searches using at least two different adapters** (e.g. `markdown_grep` lexical + `naive_rag` semantic) and also finds nothing; (3) the operator confirms the question is genuinely *answerable-shaped* (not malformed). Only when **every** search attempt fails is the query labeled negative, with the searched-adapters list recorded in the label provenance. A query for which *any* adapter surfaces a genuine supporting doc is reclassified as a positive (or discarded), never left as a negative. This prevents a missed-but-present answer from becoming a false test case that unfairly penalizes correct retrieval and inflates correct-refusal scores.

### 5.3 Difficulty bands

Target **≥ 150** labeled queries (stretch **250–300**), distributed across **five bands**:

| Band | Tests | Why it exists |
|---|---|---|
| single-hop | one doc, direct | floor; everyone should pass |
| multi-hop | answer needs ≥2 docs joined | where vector top-k tends to fail |
| contradiction / correction | corpus contains a fact and its later correction | where governance/temporal handling matters |
| recency-sensitive | most-recent value wins over stale | Zep-class territory; tests staleness handling |
| **negatives** (should-find-nothing) | answer is genuinely **not** in the corpus | tests *refusal* — see §6.5 |

**Per-band minimums (prevents degenerate distributions).** The ≥150 floor is **not** satisfiable by piling queries into one easy band. Each band carries a **minimum count**, recorded as configurable constants in `config.py` (`MIN_PER_BAND`) so downstream slices target them:

| Band | Minimum count |
|---|---|
| single-hop | ≥ 25 |
| multi-hop | ≥ 25 |
| contradiction / correction | ≥ 20 |
| recency-sensitive | ≥ 20 |
| negatives | ≥ 20 |

These floors (sum 110, leaving headroom to the ≥150 total for any band) keep every band statistically meaningful — with k=10, a 5-query band would give recall@k estimates of useless variance; ≥20 per band keeps per-band CIs interpretable. The labeling slice (s2) fails its own acceptance check if any band is below its minimum.

Band membership is a label field; metrics are reported **per band and overall**, so a system that wins only on easy single-hop cannot hide behind an aggregate.

## 6. Metrics & Exact Scoring Formulas

Notation: for a query *q*, let **G(q)** = set of gold doc-ids, **R(q)** = adapter's ranked list of returned doc-ids, **R_k(q)** = top-*k* of R(q). Default **k = 10**, pinned in config. `rel(d) = 1` if `d ∈ G(q)` else `0`.

### 6.1 Recall@k
```
recall@k(q) = |R_k(q) ∩ G(q)| / |G(q)|          (defined only for |G(q)| > 0)
```
Reported as the mean over all non-negative queries.

Note: when |G(q)| > k, recall@k has a maximum of k/|G(q)| < 1.0. Gold-set authoring (§5.2) SHOULD prefer |G(q)| ≤ k to avoid uninterpretable ceilings, or report recall@k alongside its per-query ceiling.

### 6.2 Precision@k
```
precision@k(q) = |R_k(q) ∩ G(q)| / k          (defined only for |G(q)| > 0)
```
(Denominator is **k**, not `|R_k|`; if the adapter returns fewer than k, the missing slots count as non-relevant — returning fewer docs does not inflate precision.) Reported as the mean over all **non-negative** queries (negatives have `G(q) = ∅` and are excluded here exactly as in recall@k, nDCG@k, and MRR; refusal behavior on negatives is scored once, in §6.5, never double-counted into precision).

### 6.3 nDCG@k
```
DCG@k(q)  = Σ_{i=1..k}  rel(R_i) / log2(i + 1)
IDCG@k(q) = Σ_{i=1..min(k,|G(q)|)}  1 / log2(i + 1)
nDCG@k(q) = DCG@k(q) / IDCG@k(q)        (0 if IDCG@k = 0)
```
Binary relevance (`rel ∈ {0,1}`). Reported as mean over non-negative queries. Positions i > |R(q)| are treated as non-relevant (rel=0), so DCG@k sums only over existing ranked positions when |R(q)| < k.

### 6.4 MRR (Mean Reciprocal Rank)
```
RR(q)  = 1 / rank_of_first_relevant(R_k(q))    (0 if no relevant doc in R_k(q))
MRR    = mean over non-negative queries of RR(q)
```
(Note: because the contract enforces `max_docs = k` as a hard cap, `R_k(q) = R(q)` in practice; the `R_k` notation is used here for consistency with recall@k, precision@k, and nDCG@k, so a reviewer cannot ask why MRR alone would be unbounded by the k-cutoff.)

### 6.5 Correct-refusal rate (negatives only)
For a **negative** query (`G(q) = ∅`), the *correct* behavior is to **refuse / return nothing**. **Refusal has one operational definition, drawn from the `QueryResult` contract (§3.1):** an adapter has refused on query `q` **iff** `refused == True` **AND** `ranked_results == []`. Both conditions are required — the `refused` flag is **only honored when the adapter also returns zero `ranked_results`**. This closes a credit-farming hole: an adapter that sets `refused=True` while still returning a non-empty ranked list (e.g. 10 docs on a negative) is **not** scored as a correct refusal, because returning docs is *not* refusing. (`context_string` is irrelevant to this decision — an adapter that returns `ranked_results=[]` with an explanatory `context_string` like `"No relevant documents found."` is a refusal because its ranked set is empty and it set `refused=True`; there is no separate "sentinel string" to parse.) The scoring code for `correct_refusal_rate` reads exactly these two fields and nothing else.
```
correct_refusal_rate = (# negatives the adapter refused) / (# negative queries)
false_refusal_rate   = (# positive queries where the adapter refused) / (# positive queries)
```
Positives are queries with `G(q) ≠ ∅`; "refused" is defined identically to the (corrected) correct-refusal predicate above (`refused==True AND ranked_results==[]`). `false_refusal_rate` is **reported per-adapter alongside `correct_refusal_rate`**, so a refuse-everything adapter that maxes correct-refusal is exposed by a high false-refusal rate in the same table.

**This is a scored WIN, weighted equal to a correct positive answer.** Minni's gate is *designed* to refuse when provenance is insufficient; a system that confidently hallucinates a doc for a negative query is *penalized*, not neutral. The symmetric failure — **false-refusal on a positive query** (returning empty when `G(q) ≠ ∅`) — is also tracked and reported, so "refuse everything" cannot game the negative score: a refuse-everything adapter scores 1.0 on correct-refusal but 0.0 on recall across all positive bands, and the report shows both side by side.

### 6.6 Token cost
```
token_cost(q) = number_of_tokens(context_string(q))   under the CANONICAL tokenizer
```
Reported as mean and p95 per adapter. The **canonical tokenizer is pinned** (see §7) and is the *same* tokenizer for every adapter so token cost is comparable. The harness tokenizes `context_string`; the adapter does not self-report token count.

### 6.7 Task success (Layer 2)
For an episode turn, the judge returns `success ∈ {0,1}` (and a 0–1 graded `correctness` for diagnostics). A turn is a **success** iff the agent's answer asserts the gold-fact(s) and asserts no contradicting fact. Episode success = fraction of turns successful. Per-adapter task success = mean over N trials × all episodes, reported with 95% CI.

**The headline composite is task-success-per-1k-tokens-of-retrieved-context.** The composite denominator is **context-only token cost** (the §6.6 `token_cost` over `context_string`), **not** full-prompt `tokens_to_model`. This is deliberate: `tokens_to_model` includes the shared system prompt and the episode question, which are **identical across adapters** (a single **fixed, shared system prompt** is used for every adapter in Layer 2 — see Fairness §7.12), so including them would conflate fixed prompt overhead with the thing we actually compare — retrieval efficiency. Full-prompt `tokens_to_model` is still **reported** as a separate column for total-cost transparency; the composite uses context tokens.

Both the numerator and the denominator of the composite are computed over the **same Layer-2 population — episode turns** — so the efficiency ratio never mixes the Layer-1 labeled-query set with the Layer-2 episode set:

```
mean_ctx_tokens(adapter) = mean over Layer-2 episode turns of token_cost(context_string)
                           # one context_string token count per turn (§6.6, context only),
                           # averaged across all turns in all N trials and all episodes
mean_task_success(adapter) = mean over Layer-2 episode turns of success ∈ {0,1}
                           # averaged across all turns in all N trials and all episodes
efficiency(adapter)      = mean_task_success / (max(mean_ctx_tokens, 1) / 1000)
```

(The Layer-1 §6.6 `token_cost` over the 150+ labeled queries is reported separately as a retrieval-cost metric; it is **not** the denominator of this composite. The composite is a strictly Layer-2 quantity: both terms are means over the identical set of episode turns.)

**Zero-denominator handling (no division by zero).** An adapter that returns empty `context_string` on every Layer-2 turn — realistic for Minni's gate on a negative-heavy corpus, or `native_platform` in headless mode — yields `mean_ctx_tokens = 0`. The `max(mean_ctx_tokens, 1)` **floor** keeps the formula defined. Additionally, any adapter whose `mean_ctx_tokens` rounds to 0 is **flagged and reported separately** ("returned no context — efficiency composite not meaningful"), never silently assigned an enormous efficiency score, so a refuse-everything adapter cannot top the leaderboard via a tiny denominator. This pairs with the §6.5 false-refusal penalty: such an adapter already scores 0.0 task success on positive bands, so its flagged composite is informative, not flattering. The composite is reported alongside — never instead of — the raw per-family numbers.

### 6.8 Ingest cost accounting (llm_wiki and any LLM-curated adapter)
`llm_wiki.ingest()` spends real generation tokens and wall-clock time building entity pages from the ~770-doc corpus; non-LLM adapters spend none. **Scope decision (explicit): membench's efficiency composite (§6.7) measures only QUERY-TIME context cost; one-time ingest cost is reported SEPARATELY and is NOT folded into the composite.** `IngestReport.ingest_tokens_used` and `build_wall_clock_ms` are recorded and shown in a dedicated "ingest cost" table in the report (0 for non-LLM adapters), so a critic cannot accuse the benchmark of hiding `llm_wiki`'s real build cost — it is visible, just not double-counted into a per-query efficiency number. Whether to additionally report an *amortised* ingest cost (ingest tokens ÷ total queries) is an §10 open question for s7 reporting.

### 6.9 Pre-registered comparison test (when is A better than B?)
Reporting means + 95% CIs is not enough to *claim* one adapter beats another; with N=5 trials over ~20 episodes, CIs will often overlap and "eyeballing intervals" is exactly what a reviewer attacks. membench therefore **pre-registers** the comparison procedure here in s0:
- **Unit of analysis:** per-episode success rates (mean over the N trials per episode), paired by episode across the two adapters being compared (same episodes → paired design).
- **Test:** **Wilcoxon signed-rank test** over the per-episode paired differences (non-parametric; robust to the small, non-normal episode sample and to the bounded 0–1 success metric). A paired t-test is reported as a secondary diagnostic but the Wilcoxon result is authoritative.
- **Significance threshold:** **α = 0.05**.
- **Comparison scope (fixed here to bound the FDR family):** the **confirmatory** Wilcoxon test is run on **overall task success only** — up to 5 adapters yield up to **C(5,2) = 10 pairwise comparisons**, and that is the **entire confirmatory family (size 10)**. A claim of "adapter A beats adapter B on task success" is a confirmatory claim and is made only from this overall family. **Per-band Wilcoxon tests are EXPLORATORY only:** they may be reported for insight (e.g. "Minni's edge concentrates in the multi-hop band") but are **labeled exploratory**, are **not** corrected into the size-10 confirmatory family, and **cannot** ground a headline "A beats B" claim. Per-band confirmatory testing is out of scope for the current plan because it would require each band to carry its own load-bearing paired episode subset (≥ the per-band episode floor) and would expand the family to 5 adapters × C(5,2) × 5 bands + 10 overall = 60 tests — pre-registered here as future work, not silently mixed in.
- **Multiple-comparison correction:** the 10 overall pairwise p-values are corrected with **Benjamini–Hochberg FDR** at q = 0.05 across that **size-10 confirmatory family**. A claim of "adapter A beats adapter B on task success" is only made when the FDR-corrected Wilcoxon p < 0.05; otherwise the report states "no significant difference at the current N." N (trials) and episode count are the levers to increase power and are pinned in config.

## 7. Fairness Controls (the scrutiny shield)

This is the load-bearing section. Every control below exists to kill a specific "you rigged it" objection.

1. **Identical frozen corpus bytes for every adapter.** All five ingest the *same* content-hashed snapshot. The hash is asserted equal across adapters at run start; a mismatch aborts the run. → kills *"Minni got a cleaner corpus."*

2. **Same embedding model for every vector adapter.** `naive_rag`, the semantic pass of `markdown_grep`, `llm_wiki`'s retrieval, and Minni's own vector recall all use the **one pinned embedding model id**. → kills *"Minni used a better embedder."*

3. **Same top-k and same token budget for everyone.** `k` and `TokenBudget` are global config, identical per adapter. → kills *"Minni was allowed more context."*

4. **Minni's daemon-hop + provenance-gate overhead is COUNTED AGAINST Minni, never excluded.** `wall_clock_ms` is timed *inside* `query()`, wrapping the full real daemon round-trip and gate evaluation. There is no "warm path" or "gate-bypassed" mode in the benchmark. → kills *"you only timed the fast part of Minni."* (And it is *why* we expect Minni to lose on latency — §2.)

5. **Harness lives in `bench/`, ISOLATED from `engine/` and `plugins/`.** The Minni adapter talks to Minni **only through the same public daemon interface a real client uses** — no import reaches into `engine/` internals to shortcut Minni's path (no calling `engine.principal` directly, no pre-warmed index handed in, no internal cache reach-around). A lint/CI check forbids `bench/` from importing `engine.*` or `plugins.*` private modules. → kills *"Minni's adapter cheated by skipping its own gate."*

6. **Correct refusal on negatives is a scored win, with the symmetric false-refusal penalty tracked.** (§6.5.) → kills *"Minni's refusals are just it failing, scored as success"* — because false-refusal on positives is penalized in the same report.

7. **Everything is pinned and printed.** Every run pins and records in the report header: **corpus content-hash, corpus scrub_manifest_hash, per-adapter config_hash, embedding model id, agent model id (+family), judge model id (+family), `llm_wiki` curation model id (+family), canonical tokenizer id+version, k, token budget, N trials, episode-set hash, Python version, and the metric-library lockfile hash (numpy/scipy versions).** The **episode-set hash** is defined as **SHA-256 over the sorted canonical manifest of the actual episode files consumed in the run**, computed by the runner **at run time** (not hardcoded from the fixture directory). The report header distinguishes the **"fixture episode hash"** (the public synthetic episodes, for CI reproducibility) from the **"run episode hash"** (the episodes actually consumed in a given private run, for private-run provenance); a private run records its run episode hash even though the episode bytes themselves stay under `_private/`. The runtime pin (Python + math-library versions, via the committed lockfile — §3.2, §11) is what makes the byte-identical determinism gate hold across machines. → kills *"I can't reproduce your numbers."*

8. **One canonical tokenizer, stated and pinned.** The canonical tokenizer for all token counting is **`cl100k_base` (tiktoken)**, pinned by tiktoken version in config. Rationale: it is open, deterministic, locally runnable by any reviewer, and identical across adapters, so token cost is an apples-to-apples comparison rather than a per-vendor count. (If the agent-under-test model uses a different native tokenizer, token-cost remains reported in the canonical tokenizer for cross-adapter comparability, and the native count is *additionally* reported for the agent-facing prompt; the canonical number is the one used in the §6.7 composite.) → kills *"you counted Minni's tokens with a stingier tokenizer."*

9. **Reproducible via one command from clean.** `make membench` (or equivalent single entrypoint, defined in s1) runs Layer 1 end-to-end on the synthetic fixture corpus from a clean checkout with no operator data, producing a report. The private-corpus run is the same command with a flag pointing at `_private/`. → kills *"this only runs on your laptop with secret steps."*

10. **`llm_wiki` curation LLM is pinned and capped.** The generation model that builds `llm_wiki`'s entity pages at ingest is **not** an embedding model and so is **not** covered by control 2; it has its own pin: `llm_wiki_curation_model_id` (+`model_family`) is fixed in config, printed in the report header, and **MUST equal the agent-under-test `model_id`** (CI rejects any config where they differ — see §4.1; this is the operationally enforceable form of "never stronger than the agent-under-test"). → kills *"llm_wiki cheated by pre-distilling the corpus with a frontier model nobody else had."*

11. **All adapter ingest receives only the cryptographically-verified scrubbed `FrozenCorpus`.** No adapter — embedding, lexical, or LLM-curating — ever receives raw vault bytes; `corpus.py` loads only from the scrub tool's output and cross-checks **both** `content_hash` and `scrub_manifest_hash` before instantiating a `FrozenCorpus` (a bare `scrubbed == True` boolean is not sufficient — §5.1). Report artifacts truncate logged `context_string`s to a bounded length so a missed secret cannot leak in bulk. → kills *"the llm_wiki build leaked secrets into wiki pages / context output."*

12. **One fixed, shared system prompt for all adapters in Layer 2.** Every adapter's agent runs under the **identical** system prompt and identical episode questions; only the `context_string` (wrapped in the untrusted `<retrieved_context>` boundary, §3.1) differs. This is why the §6.7 composite uses *context-only* tokens — the shared prompt is a constant, not a per-adapter variable. → kills *"adapters got different prompts, so the token comparison is apples-to-oranges."*

13. **Identical warm-up for every adapter (no Minni cold-start penalty, no FAISS warm-path advantage).** Before any *scored* query, **each** adapter runs **one un-timed warm-up query** that is discarded — this starts Minni's daemon and loads its index, loads FAISS/Chroma into memory, spawns ripgrep once, and primes `llm_wiki`'s retrieval, all outside the timed region. The first *scored* query therefore measures steady-state retrieval for every adapter, not arbitrary process-launch latency assigned to Minni alone (nor a masked startup cost for in-process adapters). The warm-up step is mandatory and ordered before scored timing; §9.9 adds a test asserting the warm-up ran before any scored measurement. **The warm-up protocol applies to BOTH Layer 1 and Layer 2.** Layer 2 does not time adapters for reporting (it tracks tokens-to-model, not `wall_clock_ms`), but cold-start can still affect *retrieval quality* for some adapters (e.g. `llm_wiki` loading its wiki into memory, an ANN index not yet mmap-resident), which would feed degraded `context_string`s into scored turns. So each adapter also runs one un-timed, discarded warm-up `query()` at the start of each Layer-2 process/run, before the first scored episode turn. → kills *"you charged Minni's daemon cold-start to its latency but let FAISS load for free"* (and its mirror image), and *"a cold adapter served worse context on the first scored Layer-2 turn."*

14. **API credentials are environment-only and never enter any artifact.** Layer 2 (agent + judge) and `llm_wiki` curation make real authenticated LLM API calls, and embedding may call a hosted model. All such credentials (API keys, bearer tokens) are sourced **only** from environment variables or a secret manager — **never** read into `config.py` at import time, **never** included in the pinned-config block printed in the report header (§7.7 pins model *ids*, not secrets), and **never** written to any file under `bench/reports/` or `bench/fixtures/`. `config.py` references credentials by env-var *name*, not value. A CI test (§9.11) scans every file under `reports/` and `fixtures/` and **fails** if any matches common credential patterns (`sk-`, `Bearer `, `api[_-]?key`, `_token`). → kills *"your report or fixtures leaked an API key."*

15. **API-cost guard (no runaway Layer-2 spend).** Because Layer 2 and `llm_wiki` curation make real authenticated LLM calls — potentially thousands across N trials × ~20 episodes × turns × adapters — the runner is rate/quota-guarded: (a) it prints a **pre-run cost estimate** (`estimated_api_calls × est_tokens_per_call`) before any call is made; (b) a **`--dry-run` flag** validates config and the full run plan **without making any API call**; and (c) a configurable **`MAX_API_CALLS`** hard-abort (in `config.py`) **kills the run** the moment the cumulative call count would exceed it. → kills *"the harness can silently rack up an unbounded API bill."*

## 8. Threats to Validity & Known Limitations

Stated brutally, because an honest benchmark names its own weaknesses before a critic does.

1. **Real-vault non-reproducibility (the big one).** The headline numbers run on a *private* corpus outsiders cannot access. Mitigation: the *pipeline* is fully reproducible on the shipped synthetic fixture corpus (anyone can verify the method); the corpus content-hash is published so the operator can prove a later run used the same bytes; the private numbers are explicitly labeled "single-operator corpus, not externally reproducible." We do **not** claim external reproducibility of the *private numbers* — only of the *method*.

2. **`native_platform` headless-driving fidelity.** Claude Code session memory / compaction is designed for an interactive session, not a benchmark harness. Driving it headlessly to expose a `query()` may not perfectly reproduce its in-session behavior, which could **understate or overstate** it. Mitigation: the adapter documents exactly how it drives the platform; this adapter's numbers carry a "fidelity-limited" flag in the report; it is explicitly the **lowest-confidence adapter**. This is an open question (§10).

3. **Synthetic episodes.** The 15–25 Layer-2 episodes are *scripted*, not harvested from real multi-month usage. They may not represent the true distribution of how the operator queries memory, and a clever author could (unconsciously) script episodes Minni happens to be good at. Mitigation: episodes are authored against the *difficulty bands* (not against any adapter's strengths), the episode set is hashed and published, and episode authoring is a separate slice (s4) reviewed for adapter-neutral phrasing.

4. **Single-operator corpus bias.** One person's vault has one person's structure, vocabulary, and note-taking habits. Results may not generalize to other users or other domains. Mitigation: stated as a scope limit — membench measures "Minni on a real personal vault," not "Minni for all users." Multi-operator corpora are future work, not claimed.

5. **LLM-judge noise.** The Layer-2 judge is itself an LLM and is imperfect, biased, and non-deterministic. Mitigation: the **dual calibration gate on a minimum-size subset** (raw agreement ≥ 0.80 **and** Cohen's κ ≥ 0.60, over a human-checked subset of **≥ 40 paired judgments** so κ's CI is not so wide that a near-chance judge slips through — κ is mandatory so a constant-answer judge cannot pass on skewed labels, §3.3), the no-self-judge gate (different `model_family`), N-trial variance reporting, the pre-registered Wilcoxon+FDR comparison test (§6.9), and publishing the judge rubric + the human-checked subset so the judge's agreement is auditable. We report both judge agreement *and* κ *as numbers in the report*, so the reader can discount accordingly.

6. **Archetype-vs-product gap.** `markdown_grep` and `llm_wiki` are faithful *reconstructions* (§4.1), not the exact products a given user runs. A real Obsidian-MCP setup or a real llmWiki install might be tuned better or worse. Mitigation: they are labeled "archetype" everywhere; the §4.1 citations justify faithfulness; Mem0/Letta/Zep are named as the products these archetypes bracket, with productized adapters flagged as future work.

7. **Latency is host-dependent.** p50/p95 depend on the machine. Mitigation: host spec (CPU, RAM, OS) recorded in the report; latency excluded from the byte-identity determinism gate (§3.2); cross-adapter latency *comparisons* are valid because all adapters run on the same host in the same run.

8. **Embedding-model choice advantages some adapters.** Whichever embedding model is pinned will suit some retrieval strategies more than others. Mitigation: the *same* embedder for all vector adapters (fairness §7.2) makes it a shared constant, not a per-adapter advantage; the choice is recorded and is an open question (§10).

9. **Prompt-injection residual risk (the boundary is mitigated, not proven).** The `<retrieved_context>` boundary (§3.1) uses a per-run nonce delimiter, literal-tag escaping, the content-only `context_string` constraint, and a system-prompt instruction — but no known wrapping fully prevents a model from being influenced by instructions embedded in retrieved data. A maliciously crafted corpus doc could in principle still steer a Layer-2 agent (e.g. toward "output SUCCESS"), corrupting task-success scoring. Mitigation/bounding: the corpus is the operator's **own** vault (§5.1), presumptively non-adversarial, which limits but does not eliminate exposure; the scrub pass and content-only constraint reduce surface; the residual is disclosed here rather than claimed away. We do **not** assert the boundary "prevents" injection — only that it is the standard mitigation and the residual is known.

## 9. How We Test the Harness Itself

The benchmark needs its own tests, or it is just assertion.

1. **Determinism test (CI gate).** Run Layer 1 twice on the fixture corpus; strip **exactly** the fields in `DETERMINISM_EXCLUDED_FIELDS = {"wall_clock_ms", "build_wall_clock_ms"}` (§3.2) and assert the remaining score artifact is **byte-identical**. The test additionally asserts (a) the excluded set is exactly those two names and (b) all score fields (`recall_at_k`, `precision_at_k`, `ndcg_at_k`, `mrr`, `token_cost`, `correct_refusal_rate`) are still present *after* the strip — so an implementer cannot pass the gate by stripping the whole record. **Meta-test — the gate must CATCH unregistered nondeterminism (new):** a fixture run emits a **third jittery field** (e.g. `run_start_epoch`) that is **NOT** in `DETERMINISM_EXCLUDED_FIELDS`; the test asserts the determinism comparison **FAILS** on that field — proving the gate detects unregistered nondeterministic fields rather than silently ignoring them (a gate that only ever checks the excluded constant would pass vacuously). Fails the build on any drift.
2. **Metric unit tests with hand-worked cases.** For tiny fixtures with known gold sets, assert recall@k, precision@k, nDCG@k, MRR, correct-refusal, and token cost equal **values computed by hand** in the test. Covers edge cases: empty gold (negatives), gold larger than k, zero relevant retrieved, adapter returns fewer than k. **False-refusal case (new):** an adapter that refuses (`ranked_results=[]`, `refused=True`) on a **POSITIVE** query (`G(q) ≠ ∅`) MUST increment `false_refusal_count` (**NOT** `correct_refusal_count`), and `recall@k` for that query MUST be **0** — asserting that a refusal on a positive is scored as a miss, never as free refusal credit.
3. **Isolation lint.** Static check that `bench/` imports no private `engine.*` / `plugins.*` module (fairness §7.5).
4. **Adapter-contract conformance test.** A shared test suite every adapter must pass:
   - `query()` returns a well-formed `QueryResult` with **no** adapter-supplied token field; the **harness-computed** token count equals `len(canonical_tokenizer.encode(context_string))`.
   - **TokenBudget enforcement (new):** the harness token count `<= budget.max_tokens` and `len(ranked_results) <= budget.max_docs`. A **negative stub** adapter that returns an over-budget `context_string` (more tokens than `max_tokens`) MUST **fail** this suite — proving an adapter cannot inflate recall by smuggling extra content past the budget.
   - `wall_clock_ms` is populated.
   - `ranked_results` contains only valid corpus doc-ids drawn from `corpus.doc_ids()`, with **unique** `doc_id`s (chunk-to-doc first-hit dedup, §3.1).
   - `context_string` is **content-only**: it contains **none** of the patterns in `contract.BANNED_ROLE_MARKERS` (the §3.1 list — `SYSTEM:`, `ASSISTANT:`, `<|system|>`, etc.), matched case-insensitively. A negative stub adapter that prepends `"Assistant:"` MUST fail.
   - `refused` is a bool; an adapter that sets `refused=True` while also returning **non-empty** `ranked_results` is allowed, but per §6.5 its `refused` flag is **not** honored as a refusal (refusal requires `refused==True AND ranked_results==[]`), so it earns **no** correct-refusal credit on negatives.
   - Includes the chunk-level adapters (`naive_rag`, `llm_wiki`, `markdown_grep` semantic pass) to prove their chunk→whole-file collapse never emits a duplicate doc.
   - **Teardown contract (new):** after `teardown()`, a call to `query()` MUST **raise** (e.g. `RuntimeError`) — it must NOT silently succeed by reusing leftover index/connection state. This proves `teardown()` actually releases resources and that stale state cannot leak into a subsequent run.
5. **Corpus-integrity + scrub-gate test.** (a) Assert all adapters report the **same corpus content-hash** at ingest (abort on mismatch). (b) Assert `corpus.py` refuses to load a corpus unless **both** `content_hash` and `scrub_manifest_hash` recompute and cross-check (a bare `scrubbed=True` Protocol object over un-scrubbed bytes is **rejected** — §5.1). (c) **Doc-count integrity (new):** for every adapter assert `ingest_report.doc_count == len(corpus.doc_ids())`; a **negative test** confirms a misconfigured adapter that ingests **0 docs** (e.g. wrong path) **aborts the run with an error**, never proceeds to a silent zero-recall result. **Over-count cross-check (new — `doc_count` is self-reported and an over-count would pass `==len(doc_ids())` falsely if the adapter returns a constant):** after ingest, spot-query for a **unique UUID string** known to be present in exactly **one** fixture doc and assert the adapter **retrieves that doc** — proving `doc_count` reflects the *actual* indexed contents rather than a hardcoded constant that happens to match `len(corpus.doc_ids())`. (d) **Path-traversal negative test (new):** `corpus.read("../../etc/passwd")` and `corpus.read("/etc/passwd")` both raise (membership + realpath-containment guard, §5.1), opening no file.
6. **Judge-gate tests (Layer 2).** Assert the harness *refuses to publish* Layer-2 numbers when (a) raw agreement < 0.80, **or (b)** Cohen's κ < 0.60, **or (c)** the human-checked subset has **fewer than 40 paired judgments** (§3.3 minimum-n gate, a hard error), **or (d)** `agent_model.model_family == judge_model.model_family`. These are negative tests: each gate must trip. **Fixture spec (concrete, so the degenerate case is actually exercised):** synthetic judge-gate subsets live in `bench/fixtures/judge_gate/` as `*.jsonl` (NOT under `_private/` — these are fabricated, not real judgments). The skewed-label degenerate fixture, `judge_gate/skewed_constant.jsonl`, contains **≥ 40** episodes of which **85% are labeled `success`**; the stub judge always returns `success`, yielding raw agreement ≈ 0.85 but κ ≈ 0 — the test asserts the κ gate **rejects** it. A second fixture `judge_gate/too_small.jsonl` has **20** episodes to assert the minimum-n gate trips. Using ≥ 40 in the degenerate fixture makes κ meaningfully non-zero-able and avoids a two-episode test that passes numerically without exercising the intent. **Fence-post boundary fixtures (new — `JUDGE_MIN_SUBSET_N=40` is an off-by-one risk):** `judge_gate/n39.jsonl` has **exactly 39** paired judgments and MUST **FAIL** the minimum-n gate, and `judge_gate/n40.jsonl` has **exactly 40** paired judgments and MUST **PASS** it (both built to clear the agreement/κ thresholds so only the n-gate is under test). The test asserts the fence-post: 39 rejects, 40 admits.
7. **Negative-control "sanity adapter".** A deliberately broken adapter (`sanity_random.py`, returns random corpus docs) must score **near-floor** on quality and **fail** task success. **Concrete thresholds (so "near-floor" is testable):** on the synthetic fixture the sanity adapter MUST satisfy `recall@k < 0.10`, `ndcg@k < 0.05`, `mrr < 0.05`, and `task_success < 0.15`. If any metric returns a higher value for random docs, the metric is mis-wired and the test fails. (Thresholds live as constants in the test; they are set well above pure-random expectation for the fixture's label density but far below any real adapter, giving an unambiguous oracle.) **Minimum fixture size (so the floors are not vacuously met on a tiny corpus):** the synthetic fixture exercised here MUST contain **at least 50 docs** and **≥ 25 labeled queries** with **≤ 3 gold docs per query**, so the random-baseline floors (`recall@k < 0.10`, `ndcg < 0.05`, `mrr < 0.05`, `task_success < 0.15`) hold **in expectation** rather than passing by accident on a handful of docs where a random pick is likely to be relevant.
8. **Private-tree gitignore test (`test_private_gitignored.py`).** Runs `git check-ignore` on representative `_private/` paths (e.g. `_private/membench/corpus_real/x.md`, `_private/membench/gold_real.jsonl`) and **fails if any is not ignored**. Machine-checks the §5.1 invariant so a removed/incorrect `.gitignore` rule cannot silently stage private vault data during later-slice scaffolding.
9. **Warm-up-ordering test.** The runner emits a **structured trace** of ordered events per adapter (`WARMUP_START`, `WARMUP_DONE`, `SCORED_QUERY_START`, …). The test asserts that for **every** adapter a `WARMUP_DONE` event appears **before** the first `SCORED_QUERY_START` event, and that the warm-up result is **not** in the scored records (its `query_index` is absent from the score artifact). This gives the test a concrete observable (the trace event order) rather than internal-state coupling, so it cannot pass vacuously by merely checking that scores exist. **Completeness assertion (new — the ordering check is vacuously true if the runner crashes after `WARMUP_DONE` and emits no scored query):** for **each** adapter the test additionally asserts that **at least one `SCORED_QUERY_START` event exists**, so "no `SCORED_QUERY_START` before `WARMUP_DONE`" can never be satisfied trivially by a crashed run that never scored anything. Applies to both Layer-1 and Layer-2 warm-ups (fairness §7.13).
10. **Subprocess-safety lint (`test_subprocess_safety.py`).** AST static check over **every** file under `bench/` (not just `markdown_grep`), enforcing **two concrete syntactic rules** (the earlier "derived from `q`" phrasing was untestable — taint-tracking a variable to `q` is undecidable in general; these are decidable AST checks instead): **(1) ban `shell=True`** — no call may pass `shell=True` to any `subprocess.*` API; **(2) ban string-built argv** — no `subprocess.*` argument list may contain an element produced by **f-string, `%`-formatting, or `+` string concatenation** (i.e. every argv element must be a literal or a bare name, never an interpolated/concatenated string expression). Together these structurally prevent any caller-controlled value (including `q`) from being interpolated into a subprocess argument. A future adapter or maintenance edit that reintroduces `shell=True` or a string-built argv fails CI. This machine-enforces the §3.1/§4.1 command-injection prohibition that previously lived only in prose.
11. **Credential-leak scan (`test_no_secrets_in_artifacts.py`).** Scan every file under `reports/` and `fixtures/` and **fail** on any match of common credential patterns (`sk-`, `Bearer `, `api[_-]?key`, `_token`); also assert `config.py` reads no credential value at import time (it references env-var names only). Machine-enforces fairness §7.14 so a key cannot leak into a published report or a committed fixture. **Known limitation:** this scan matches plaintext credential patterns only; it may **miss base64- or URL-encoded keys** (an encoded `sk-…` would not match `sk-`). This is a known, accepted gap — the env-only-credentials control (§7.14) is the primary defense; the scan is a backstop.
12. **Context-string log-truncation test (`test_context_log_truncation.py`).** Injects a **10,000-char** `context_string` (and a 10,000-char `FrozenCorpus.read()` result) through the code paths that may log them, captures all emitted log records at DEBUG+, and **asserts no emitted log line exceeds `CONTEXT_LOG_TRUNCATE` + a small margin**. Machine-enforces the §5.1 single-log-safe-wrapper rule so a missed secret cannot leak in bulk via any DEBUG log path before `report.py`.

## 10. Open Questions

These are unresolved and must be decided in named downstream slices, not silently in code:

1. **Embedding model choice.** Which single embedding model is pinned for all vector adapters? (Trade-off: local/open vs strongest-available; affects fairness §7.2.) → decide in s1.
2. **Judge vs agent model pairing.** Which concrete (agent_model, judge_model) pair satisfies the different-family no-self-judge gate while keeping the judge strong enough to clear 0.80 agreement? → decide in s5 (judge calibration). Per the Anthropic-model triage rule, if the agent or judge is a Claude model, confirm current model ids/pricing against live docs at that slice rather than from memory.
3. **Canonical tokenizer.** This spec pins **`cl100k_base`** as the cross-adapter counting tokenizer (§7.8). Open sub-question: do we *also* report each agent model's native token count as a secondary column? (Leaning yes, for honesty; confirm in s7 reporting.)
4. **`native_platform` fidelity.** Can Claude Code session memory be driven headlessly with enough fidelity to be a fair adapter, or must it carry a permanent "fidelity-limited" asterisk (or be dropped from headline numbers)? → resolve in s3 (native_platform adapter).
5. **Gold-set size.** Land at the **≥150** floor or push for the **250–300** stretch before publishing? Trade-off: operator review cost vs statistical power per band (esp. the negatives and contradiction bands, which need enough samples to be meaningful). → decide in s2 (corpus & labeling).
6. **Productized Mem0 / Letta / Zep adapters.** Add as first-class adapters post-s7, or leave bracketed by archetypes? → future-work decision, out of scope for the current 8-slice plan.
7. **Amortised ingest cost.** Should the report additionally show `llm_wiki`'s ingest token cost amortised per query (ingest tokens ÷ total queries) alongside the separate ingest-cost table (§6.8)? It stays OUT of the §6.7 composite either way. → decide in s7 (reporting).
8. **Runtime-in-session writes.** The current plan tests recall of the pre-ingested frozen corpus only; in-session writes (agent persists a new fact mid-episode, retrieves it later — §3.3) are deferred because they require a separate fairness design for stateless adapters. → future work, out of scope for the current 8-slice plan.
